# DDD とは

## 概要

DDD は Domain-Driven Design の略で、日本語では「ドメイン駆動設計」と呼ばれます。  
ここでいうドメインは、アプリが扱いたい業務やルールのことです。

たとえばこの学習用アプリなら、

- 記事は下書きから公開へ進む
- 公開済み記事は再度公開できない
- 公開予定日は過去にできない

のような「記事投稿という題材ならではのルール」がドメインです。

DDD は、そうした業務ルールをコードの中心に置いて、意味のある名前と構造で表現しようとする考え方です。

## DDD で大事にすること

- 技術都合より先に業務の意味を考える
- ルールを Controller に散らさず、意味のある場所に集める
- 名前を業務の言葉に寄せる
- データだけでなく振る舞いもモデルに持たせる

## よく出てくる基本用語

### ドメイン

アプリが解決したい業務領域のことです。  
このプロジェクトなら「記事投稿」がドメインです。

### エンティティ

同じものとして追跡したい対象です。  
ID を持ち、状態が変わっても同じ存在として扱います。

例:

- `Article`

### 値オブジェクト

値そのものに意味がある小さなオブジェクトです。  
同じ値なら同じ意味として扱います。

候補例:

- `PublishDate`
- `ArticleTitle`

### 振る舞い

「何ができるか」を表す処理です。  
ただデータを持つだけでなく、その対象自身のルールを表します。

例:

- `Article::publish()`

### ユースケース

ユーザーがやりたいことを1つの処理としてまとめたものです。

例:

- 記事を投稿する
- 記事を公開する

Laravel で進めるなら、後のフェーズで `CreateArticleUseCase` や `PublishArticleUseCase` のような形に切り出していくイメージです。

## Laravel で最初に起きやすいこと

Laravel では最初、Controller に処理を書くだけでも十分動きます。  
ただし機能が増えると、次のようなつらさが出やすくなります。

- 条件分岐が Controller に増える
- 文字列ルールがあちこちに散る
- 何が業務ルールで、何がフレームワーク都合かが混ざる

DDD は、そのつらさが見えてきたときに整理する助けになります。

## このプロジェクトでの DDD の入り口

今のコードでは、すでに少しずつ DDD 的な形へ寄せています。

- `status` を `ArticleStatus` enum にしている
- 公開ルールを `Article::publish()` に寄せている
- HTTP の処理と業務ルールを少しずつ分け始めている

つまり今は、最初から純粋な DDD を目指しているのではなく、

1. まず普通に動かす
2. テストで仕様を固定する
3. 苦しくなった場所を少しずつ整理する

という実践寄りの進め方をしています。

## 今後の学習で伸ばしたいポイント

- `publish_date` を単なる日付ではなく意味のある値として扱う
- `publish()` から保存処理を分離する
- 例外を業務の言葉で表す
- UseCase を導入して Controller を薄くする

## ひとことで言うと

DDD は、「業務の意味が伝わる形でコードを整理していく考え方」です。  
この学習では、記事投稿アプリを題材にしながら、その入口を少しずつ体験していきます。
