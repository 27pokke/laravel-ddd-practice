# Value Object とは

## 概要

Value Object は、DDD でよく使われる「意味のある値」を表す小さなオブジェクトです。  
ただの文字列や日付のままだと曖昧になりやすい値に、名前とルールを与えるために使います。

このプロジェクトなら、たとえば次のようなものが Value Object の候補です。

- `PublishDate`
- `ArticleTitle`

どちらも単なる値ではなく、

- 公開予定日として正しいか
- 記事タイトルとして正しいか

という意味やルールを持っています。

## 役割

Value Object の主な役割は、値に意味を持たせることです。

- 業務の言葉で値を表現する
- 値に関するルールを1か所へ集める
- 不正な値を作りにくくする
- `string` や `date` のまま扱うより読みやすくする

たとえば `publish_date` がただの文字列だと、

- どんな形式を期待しているのか
- 過去日付を許すのか
- その値が何を意味しているのか

がコード上でぼやけやすくなります。

`PublishDate` という型にすると、

- これは公開予定日である
- 過去日付は許さない

という意味をコードで表せます。

## Entity との違い

Entity は ID を持ち、同一性で扱うものです。  
一方で Value Object は、値そのもので扱います。

例:

- `Article` は Entity
- `PublishDate` は Value Object

`Article` は同じ ID なら同じ記事ですが、`PublishDate` は同じ日付なら同じ意味として扱います。

## Value Object の特徴

### 値そのものに意味がある

Value Object は「何の値なのか」が大事です。  
同じ文字列でも、ただの文字列なのか、記事タイトルなのか、メールアドレスなのかで意味が違います。

### 不正な状態を作りにくい

Value Object を作るときにルールを通すことで、不正な値を早い段階で防げます。

例:

- `PublishDate` で過去日付を弾く
- `ArticleTitle` で空文字や長すぎるタイトルを弾く

### 小さいが重要

Value Object は地味ですが、設計をかなり分かりやすくします。  
大きなクラスではなくても、コードの意味を明確にする役割があります。

## このプロジェクトでの使いどころ

今の学習では、すでに `PublishDate` を Value Object として使い始めています。

これは、

- 公開予定日という値に意味を持たせる
- 過去日付は不可というルールをまとめる
- Controller の validation だけに頼りすぎない

ためです。

たとえば `CreateArticleUseCase` で文字列をそのまま保存するのではなく、いったん `PublishDate` を通してから記事生成へ渡すと、コードの意図が読み取りやすくなります。

## Value Object を使うメリット

- コードを読んだときに意味が伝わりやすい
- ルールの置き場所がはっきりする
- 同じチェックを何度も書かずに済む
- Domain に寄せた設計にしやすい

特に学習では、「Controller に全部書く」状態から一歩進んで、「値にも責務を持たせる」感覚をつかみやすいです。

## 気をつけたいこと

- 何でも Value Object にすればよいわけではない
- 小さくても意味がある値に絞る
- DB に保存するときの値と、ドメイン上の型は分けて考える

たとえば `PublishDate` はドメイン上では Value Object ですが、DB に保存するときは文字列や日付型に変換して扱う必要があります。

## ひとことで言うと

Value Object は、「ただの値に、意味とルールを与えるための型」です。  
DDD では、業務の言葉をコードに落とし込むための大事な道具の1つです。
