# TDD とは

## 概要

TDD は Test-Driven Development の略で、日本語では「テスト駆動開発」と呼ばれます。  
ざっくり言うと、実装の前か、少なくとも実装と同じくらい早い段階でテストを書いて、仕様を先に固定していく進め方です。

TDD の目的は、単にテストを増やすことではありません。  
「このコードは何を満たせば正しいのか」を先に言葉にして、安心して実装や改善を進められるようにすることです。

## TDD で大事にすること

- まず仕様をテストで表現する
- テストが通る最小限の実装をする
- 通ったあとで設計を整理する

この流れはよく、

- Red: 失敗するテストを書く
- Green: 通る最小限の実装をする
- Refactor: テストが通ったまま整理する

と説明されます。

## このプロジェクトでの TDD の役割

この学習用アプリでは、TDD を「仕様を固定するための道具」として使っています。

たとえば、

- 今日の公開予定日なら記事を投稿できる
- 過去日付の公開予定日では投稿できない
- 下書き記事は公開できる
- 公開済み記事は再度公開できない

のような仕様を、まずテストとして書いています。

これにより、

- 実装を変えても壊れていないか確認しやすい
- 何が要件なのかがコードから読み取りやすい
- 後で DDD 的に整理するときも安心してリファクタできる

というメリットがあります。

## Feature Test と Unit Test の違い

このプロジェクトでは、2種類のテストを使い分けています。

### Feature Test

HTTP リクエストを通して、ユーザーから見た動きを確認します。

例:

- `/articles` に投稿したら記事が作られる
- 公開 API を叩いたら記事が公開される

見るもの:

- ルーティング
- バリデーション
- Controller の動き
- DB 保存結果

### Unit Test

モデルや小さなクラスの振る舞いだけを直接確認します。

例:

- `Article::publish()` を呼ぶと状態が公開になる
- すでに公開済みなら例外になる

見るもの:

- 業務ルール
- 状態遷移
- 小さな単位の責務

できるだけ HTTP や DB から切り離して考えるのがポイントです。

## TDD の進め方の基本

1. まず仕様を1つ決める
2. その仕様を失敗するテストで書く
3. 通る最小限の実装をする
4. テストが通ることを確認する
5. 設計を少し整える

たとえば「公開済み記事は再度公開できない」を進めるなら、

1. 例外になるテストを書く
2. いったん失敗を確認する
3. `Article::publish()` に条件を入れる
4. テストを通す
5. 必要なら例外クラス名や責務分担を整理する

という流れになります。

## TDD の良いところ

- 仕様が明文化される
- 安心してリファクタしやすい
- バグに気づきやすい
- どこに責務を置くべきか考えやすくなる

特にこのプロジェクトでは、TDD があることで DDD に寄せる作業がかなりやりやすくなります。

## 気をつけたいこと

- テストを書くこと自体が目的にならないようにする
- 実装の細かい中身を固定しすぎない
- 1つのテストで確認したいことを増やしすぎない

大事なのは、「どう実装したか」より「何を満たすべきか」を表現することです。

## ひとことで言うと

TDD は、「先に仕様をテストで決めてから、安心して実装と改善を進めるための進め方」です。  
この学習では、Feature Test と Unit Test を使い分けながら、その感覚を身につけていきます。
